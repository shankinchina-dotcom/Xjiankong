# Xjiankong v2-beta 策划：AI 情报终端与历史版本浏览

> 状态：策划草案。历史与数据实施尚待闸门确认；“深空研判报告”前端视觉方向已于 2026-07-10 经老板确认，详见 `2026-07-10-v2-beta-deep-space-report-ui-design.md`。不改代码、不部署、不访问 NAS、不触发 AI。
>
> 基线：v2-alpha 已于 2026-07-09 生产上线，当前生产镜像为 `xjiankong-trendradar:v2-alpha-20260709`。
>
> 目标版本：v2-beta。
>
> 范围：页面信息架构、前端 UI、历史版本浏览、最小必要的静态索引生成。新增数据源、数据库 schema 变更、Cloudflare/NAS 操作不在本策划范围。

## 一、背景与问题

v2-alpha 已完成「AI 行业每日研判报告」8 板块升级，解决了报告从新闻聚合到研判分析的问题。但当前页面仍有四个明显短板：

1. **页面结构仍偏信息流**：RSS、热榜、AI 研判之间的主次关系还不够明确，阅读体验容易回到「刷列表」。
2. **历史快照有底层能力，但没有入口**：TrendRadar 每次运行都会生成 `output/html/YYYY-MM-DD/HH-MM.html`，生产定时任务按每 4 小时运行，一天可能有多版；但页面没有日期和版本选择器。
3. **RSS/热榜与研判报告视觉割裂**：当前已有局部视觉优化，但整体还没有形成统一的「AI 情报终端」风格。
4. **原始消息链接 404**：当前从页面点开某条消息，想追踪到原始页面时会出现 404。这会破坏 SourceRef/source_map 的可信溯源闭环，必须作为 v2-beta 前置修复项处理。

这次 v2-beta 的目标不是继续堆新闻，而是把页面升级成一个可回看、可追溯、可快速判断变化的日报终端。

## 二、产品目标

把 `trend.shankluo.cc` 从「最新日报页面」升级为：

> **AI 情报终端：默认展示最新研判，同时可以按日期和时间回看每一版历史报告。**

更长期的终极目标是：把它建设成老板个人的 **AI 情报收集中心**。收录范围不再局限于纯 AI 技术方向，而是覆盖：

- AI 基础技术：模型、论文、Agent、开源项目、开发者生态。
- AI 应用技术：AI 如何进入内容、销售、教育、电商、视频、音乐、小说、漫剧、游戏、办公自动化等具体场景。
- AIGC 创业方向：漫剧、AI 小说、AI 音乐、新兴内容生产工具和新的 AI 创业机会。
- AI 成本与商业化：主流模型价格、API 成本、降价活动、免费额度、商业化案例。
- 平台与政策环境：内容平台反应、版权规则、审核政策、分发机制、监管和合规变化。

用户应能完成四类动作：

1. **看现在**：打开首页后，3 分钟内读完当前最新 AI 行业核心判断。
2. **看过去**：选择昨天、前天或任意已有日期，打开对应版本。
3. **看一天内多版变化**：同一天内按时间线查看 00/04/08/12/16/20 等版本，知道当前看到的是哪一版。
4. **能追溯来源**：点击报告中的消息、RSS、引用来源时，应能打开可用的原始消息或合理 fallback 页面，不应普遍出现 404。

后续内容板块还需预留两类能力：

1. 「主流模型官方价格动态」：用于跟踪 Claude、OpenAI GPT、Gemini、Grok、智谱 GLM、DeepSeek、通义千问、GPT Image 2、字节 Seedream 等主流模型与图像模型的官方价格和活动变化。
2. 「AI 创业方向与 AIGC 应用风向」：用于跟踪 AIGC / AI 创业项目的热门新闻、平台反应、商业化趋势和不同创业板块的数据变化。

该终极目标不会一次性全部实现。v2-beta 第一阶段仍先完成历史版本浏览和整体 UI 升级；价格动态、AIGC 创业方向和具体应用赛道作为后续内容能力进入设计队列。

## 三、设计原则

### 3.1 研判优先，新闻作为证据层

页面首屏仍以「AI 行业每日研判报告」为核心。RSS、热榜和原始来源不抢主位，只作为证据与补充情报。

### 3.2 历史是产品能力，不是文件目录

不打开 Nginx `autoindex`，不暴露 `/html/` 目录列表。历史浏览必须通过受控页面或受控 JSON manifest 呈现，只展示允许访问的 HTML 报告路径。

### 3.3 只展示真实数据，不伪造趋势

v2-beta 可以展示：

- 报告生成时间
- 当日报告版本数
- 每版报告路径
- 热榜命中、RSS 命中、AI 分析是否存在
- 当前版本是否为最新

v2-beta 暂不展示：

- 信号强度分数
- 熵值
- 升温/冷却判断
- 话题簇图
- 昨日差分

这些指标需要额外上游数据支持，不能只靠 UI 编造。

### 3.4 不改变 v2-alpha 的 8 板块研判框架

历史浏览和新 UI 是外壳升级，不替代 v2-alpha 的核心分析结构。

但 v2-beta 的长期方向会扩大情报解释范围：同一条新闻不仅判断其对 AI 技术生态的意义，也要判断其对 AI 应用、内容生产、创业机会、平台政策和成本结构的影响。扩展的是分析视角，不是无边界扩大新闻噪音。

8 板块继续保持：

1. 今日核心判断
2. 研究/模型突破
3. 产品与工具变化
4. 开源项目与开发者生态
5. 大厂/融资/政策/算力动态
6. X 关键人物观点
7. 风险与下周关注
8. 原始来源索引

### 3.5 溯源链接必须可验证

v2-alpha 的 SourceRef/source_map 目标是让报告结论可以回到原始消息。若页面链接点击后普遍 404，报告可信度会下降。

v2-beta 必须增加一个前置修复项：

- 先区分 404 来源：URL 生成错误、Nitter 实例不可用、源站删除、平台登录限制或历史快照链接失效。
- 对 Nitter/X 来源，优先评估是否需要把展示链接从 `nitter.net/<handle>/status/<id>` 转换为 `x.com/<handle>/status/<id>`，或同时保留「Nitter 镜像 / X 原文」两个入口。
- 对 RSS、GitHub、HuggingFace、论文来源，保留原 feed 中的 canonical URL，不用 AI 生成链接。
- 生成报告时不把无法解释的 404 链接静默当作正常来源；至少在本地验证阶段抽样检查链接结构。
- 不因为链接 404 删除 source_id；应修复可点击目标或提供 fallback。

## 四、目标信息架构

### 4.1 页面整体结构

```text
┌──────────────────────────────────────────────┐
│ 顶部情报栏：项目名 / 当前版本时间 / 历史入口   │
├──────────────────────────────────────────────┤
│ 今日研判 Hero：一句话判断 + 关键指标           │
├──────────────────────┬───────────────────────┤
│ 8 板块研判报告主体    │ 右侧历史与证据侧栏      │
│                      │ - 当前日期             │
│                      │ - 当天版本时间线        │
│                      │ - 最近 7 天入口         │
│                      │ - 来源索引快捷入口      │
├──────────────────────┴───────────────────────┤
│ 过去 24 小时精选情报 / RSS 证据卡片流          │
├──────────────────────────────────────────────┤
│ 原始来源索引 / 全部来源                        │
└──────────────────────────────────────────────┘
```

### 4.2 首页默认行为

访问 `https://trend.shankluo.cc/` 时：

1. 默认展示最新报告。
2. 顶部明确显示：
   - 当前版本：`2026-07-09 20:00`
   - 是否最新：`最新版本`
   - 今日已有版本数：例如 `今日 6 版`
3. 提供「历史版本」入口，打开右侧抽屉或独立历史页。

### 4.3 历史版本入口

历史入口采用两层结构：

```text
历史版本
├── 今天 2026-07-09
│   ├── 20:00 最新
│   ├── 16:00
│   ├── 12:00
│   ├── 08:00
│   └── 04:00
├── 昨天 2026-07-08
│   ├── 20:00
│   ├── 16:00
│   └── 12:00
└── 更早
    ├── 2026-07-07
    └── 2026-07-06
```

点击任一时间版本后，打开对应静态 HTML：

```text
/html/2026-07-09/20-00.html
```

如果当前报告文件名使用实际分钟，例如 `20-51.html`，则 UI 展示为 `20:51`，不强行归并为整点。

## 五、历史数据呈现设计

### 5.1 历史侧栏

历史侧栏放在桌面端右侧，移动端改为顶部按钮打开抽屉。

内容：

| 区域 | 说明 |
|---|---|
| 当前版本 | 日期、时间、是否最新 |
| 今日时间线 | 当天所有版本，倒序排列 |
| 最近 7 天 | 每天显示版本数和最新时间 |
| 快捷操作 | 复制当前版本链接、回到最新版 |

视觉规则：

- 当前版本使用高亮边框。
- 最新版本使用低噪声 badge，不用刺眼红色。
- 历史版本使用淡蓝渐变底色，保持和 AI 研判区域统一。
- 不使用目录树式文件浏览 UI，避免像后台管理页。

### 5.2 历史独立页

新增一个独立历史页更稳，建议路径：

```text
/history.html
```

页面结构：

```text
┌──────────────────────────────────────┐
│ AI 情报历史归档                       │
│ 最近生成：2026-07-09 20:51            │
├──────────────────────────────────────┤
│ 日期筛选：今天 / 昨天 / 最近 7 天 / 全部 │
├──────────────────────────────────────┤
│ 2026-07-09                            │
│ 20:51 最新 · 热榜 11/11 · RSS 38/44    │
│ 16:32 · 热榜 11/11 · RSS 36/44         │
│ 12:03 · 热榜 11/11 · RSS 40/44         │
├──────────────────────────────────────┤
│ 2026-07-08                            │
│ 20:51 · v2-alpha 验证报告              │
└──────────────────────────────────────┘
```

独立页的价值：

- 不挤占日报正文。
- 可直接收藏。
- 更适合以后做日期搜索、版本对比和归档清理。

### 5.3 报告内历史提示

每一份历史报告顶部都应显示当前上下文：

```text
当前查看：2026-07-09 16:32 版本
这不是最新版本 · 查看最新
```

当用户打开旧版本时，避免误以为看到的是当前最新报告。

### 5.4 一天多版的表达方式

一天内多版不要写成「第 1 版、第 2 版」，因为手动触发会打乱顺序。建议使用真实时间：

```text
2026-07-09
20:51
18:32
16:32
12:03
08:02
04:01
```

如需标识定时/手动，必须有真实运行来源字段；当前没有可靠字段时不展示。

## 六、后续内容板块：主流模型官方价格动态

该板块是后续内容升级方向，不进入第一轮历史 UI mockup 的硬性交付范围。加入原因是模型价格变化会直接影响 AI 产品、Agent、内容创作和开发工具的成本判断，适合作为每日研判报告的固定观察维度。

### 6.1 目标

围绕主流模型的官方价格与活动变化，回答三个问题：

1. 最近几个主流模型是否发生降价、促销、免费额度调整或套餐变化。
2. 降价幅度或单位成本变化是多少。
3. 相比同类模型，它的价格优势、适用场景和潜在影响是什么。

### 6.2 初始关注对象

第一批关注对象：

- Claude
- OpenAI GPT 系列
- Gemini
- Grok
- 智谱 GLM / GLM 5.2
- DeepSeek
- 通义千问 / Qwen
- GPT Image 2
- 字节 Seedream

后续新增模型必须有明确官方价格页、官方公告、开发者文档或公开 API 定价入口，不能只依据二手传闻。

### 6.3 展示形式

建议在 v2-beta 或后续版本中以独立板块呈现：

```text
主流模型价格动态
├── 今日价格变化
│   ├── DeepSeek：输入价下降 xx%，输出价下降 xx%
│   └── Gemini：新增免费额度 / 套餐变化
├── 横向对比
│   ├── 文本模型：每百万 tokens 输入/输出价格
│   ├── 推理模型：推理成本与上下文长度
│   └── 图像模型：单张图生成/编辑成本
└── 价格优势判断
    ├── 最适合低成本批处理
    ├── 最适合高质量推理
    └── 最适合图像生成/编辑
```

UI 上不建议做复杂财务表格。首页只展示 3-5 条高价值变化；完整价格矩阵可放到可折叠详情或独立对比页。

### 6.4 数据与可信度约束

价格信息高度时效且容易误导，必须满足以下约束：

- 只引用官方价格页、官方公告、官方开发者文档或官方 API 返回。
- 不使用社媒截图、二手博客、转述文章作为最终价格依据。
- 每条价格变化必须记录来源 URL 和抓取/核验时间。
- 价格单位必须统一，例如 `USD / 1M input tokens`、`USD / 1M output tokens`、`USD / image`。
- 必须区分标准价、限时活动价、区域价、企业合约价和免费额度。
- 不把 AI 模型自行生成的价格写入报告；AI 只负责总结已核验数据。

### 6.5 与现有 8 板块关系

第一阶段不改 v2-alpha 8 板块结构。价格动态可以先放入：

- `产品与工具变化`：当价格变化来自模型/API 产品调整。
- `大厂/融资/政策/算力动态`：当价格变化反映算力成本、竞争策略或大厂补贴。

如果价格信息稳定出现，再升级为独立第 9 板块或「价格动态」证据层。该升级需要另起设计和实施计划。

## 七、后续内容板块：AI 创业方向与 AIGC 应用风向

该板块用于跟踪 AIGC 和 AI 应用创业方向的热门新闻、平台动作、用户需求变化和商业化机会。它不是泛泛罗列创业新闻，而是围绕具体创业赛道做趋势研判。

### 7.1 目标

围绕不同 AI 创业板块回答四个问题：

1. 最近有什么高关注新闻、产品或平台动作。
2. 该板块当前处于爆发、验证、降温还是分化阶段。
3. 有哪些可量化数据支持判断。
4. 对个人创业者、小团队或内容创作者意味着什么。

### 7.2 初始关注板块

第一批关注三个方向：

#### 漫剧板块

关注内容：

- 真人短剧、动漫短剧、AI 漫剧、AI 剧集的趋势变化。
- 抖音、快手、B 站、视频号、小红书、海外短视频平台等对 AI 剧内容的推荐、审核、扶持或限制反应。
- AI 生成角色、分镜、配音、背景、剪辑在短剧生产链条中的采用情况。
- 平台对「AI 制作」「AI 辅助制作」「搬运/低质混剪」的态度变化。

可观察数据：

- 爆款 AI 剧播放量、互动量、完播率、评论情绪。
- 平台榜单、话题热度、关键词搜索趋势。
- 创作者数量、MCN/工具方动作、商业化案例。
- 真人剧与动漫剧在制作成本、上线周期、爆款概率上的差异。

#### AI 小说板块

关注内容：

- AI 辅助网文、互动小说、角色扮演剧情、短篇故事生成工具。
- 番茄、起点、知乎盐选、飞卢、海外 WebNovel / Wattpad 等内容平台的 AI 内容政策与实际反应。
- AI 小说从「写作辅助」到「自动生成内容资产」的边界变化。
- 小说 IP 与漫剧、短剧、游戏、播客、有声书之间的联动机会。

可观察数据：

- AI 小说相关工具搜索量、用户增长、付费转化案例。
- 平台规则、封禁案例、征文活动或 AI 辅助声明要求。
- 爆款题材、更新频率、内容质量争议。
- 小说 IP 二次开发成本和转化链路。

#### AI 音乐板块

关注内容：

- AI 作曲、AI 翻唱、AI 歌手、AI BGM、短视频配乐、游戏音乐生成。
- Suno、Udio、国内音乐生成工具及平台版权政策变化。
- 抖音、B 站、小红书、Spotify、YouTube 等平台对 AI 音乐的分发、版权和标注要求。
- AI 音乐在短视频、广告、游戏、播客、虚拟人和个人 IP 中的商业化使用。

可观察数据：

- AI 音乐作品播放量、收藏量、二创数量。
- 工具价格、生成质量、版权条款和商用限制。
- 平台下架、标注、版权投诉和授权案例。
- 创作者使用 AI 音乐后的生产效率和成本变化。

### 7.3 新兴 AI 创业方向池

除三个初始方向外，后续可维护一个「新兴 AI 创业方向池」，只记录有连续信号的方向：

- AI 视频广告
- AI 虚拟主播 / 虚拟人
- AI 电商素材生成
- AI 教育陪练
- AI 销售 Agent
- AI 情感陪伴
- AI 游戏资产生成
- AI 个人知识管理
- AI 工作流自动化

进入正式板块的条件：

1. 至少连续 3 次报告出现相关信号。
2. 有可核验的数据或平台动作。
3. 能说明商业化场景，而不只是概念热。

### 7.4 数据与可信度约束

该板块必须区分「新闻热度」和「创业机会」。单条爆款新闻不能直接推导为机会。

可信来源优先级：

1. 平台官方规则、公告、创作者政策、API 或商业化政策。
2. 平台榜单、公开播放量、互动量、搜索指数、应用商店排名。
3. 创业公司官方发布、融资信息、产品更新、价格页。
4. 媒体报道和社媒讨论只能作为线索，不能作为最终判断。

必须避免：

- 不核验就说某赛道爆发。
- 只看播放量，不看商业化和平台政策。
- 把工具发布等同于创业机会。
- 把 AI 生成内容数量等同于用户需求。

## 九、技术方案

### 9.1 最小落地方案：静态历史 manifest

新增静态索引文件：

```text
output/history.json
output/history.html
```

`history.json` 示例：

```json
{
  "generated_at": "2026-07-09T20:51:00+08:00",
  "latest": {
    "date": "2026-07-09",
    "time": "20-51",
    "path": "/html/2026-07-09/20-51.html"
  },
  "days": [
    {
      "date": "2026-07-09",
      "versions": [
        {
          "time": "20-51",
          "path": "/html/2026-07-09/20-51.html",
          "is_latest": true
        }
      ]
    }
  ]
}
```

第一版只要求记录：

- `date`
- `time`
- `path`
- `is_latest`

不要在第一版解析 HTML 内容提取复杂指标。指标可以后续从已有 `report_data` 安全写入。

### 9.2 为什么不用 Nginx 目录列表

不建议开启 `/html/` 目录列表，原因：

1. 安全边界更差，容易暴露不该展示的文件名。
2. 视觉不可控，无法做产品化历史浏览。
3. 与当前 `/.env`、`/news/test.db` 404 的安全策略不一致。

### 9.3 为什么不改数据库 schema

历史报告本来就是 HTML 文件快照。v2-beta 的历史浏览第一版只需要索引这些文件，不需要新增数据库表。

数据库 schema 变更会带来迁移、回滚和生产风险，不适合这次 UI 升级。

### 9.4 source_map 约束

继续保持 v2-alpha 约束：

- `SourceRef/source_map` 只在单次运行时存在。
- 不把 `source_map` 写入 SQLite。
- 历史报告 HTML 内已渲染出的来源链接可以保留。
- 历史 manifest 不记录完整 source_map。

### Gate 3：原始消息链接 404 只读诊断

状态：**已完成（2026-07-10，只读）**。

审计结果：

- 已在本地历史报告抽样复现一条热榜外链 404：`https://wallstreetcn.com/articles/3776526`。报告确实将该外部目标作为可点击原始链接展示。
- 抽样 Nitter 链接与对应 `x.com/<handle>/status/<id>` HTTP 层均为 200；这不能证明正文一定无需登录，但说明本次样本的 404 不是本地 HTML 路由导致。Nitter 仍是用户点击链路的单点外部风险，因为页面没有 X fallback。
- GitHub canonical 样本 HTTP 200；知乎样本为 HTTP 403，属于当前诊断客户端的访问／反爬限制，不能据此断言普通用户浏览器一定不可访问。
- `SourceRef/source_map`、AI 来源索引和正文引用均原样采用外部 URL，只做协议白名单与 HTML 转义。当前普通热榜渲染与 SourceRef 都采用 `url → mobileUrl`，未发现两条路径优先级不一致。
- 未发现本地 `/news/...`、相对路径或历史快照路由是本样本外链 404 的直接原因。

结论：下一阶段应实现“来源 URL 可用性与 fallback 呈现策略”，而不是误修本地路由。对 Nitter/X 评估双入口或 X fallback；对已确认源站 404 的条目保留 `source_id`，改为不可点击提示或明确“原始页面当前不可用”；对受限站点保留 canonical URL 并标示可能的登录／访问限制。

目标：在改 UI 前先确认当前生产/本地报告中原始消息链接 404 的原因。

允许：

- 本地读取已生成 HTML。

- 改 NAS。
- 改 Cloudflare。
- 重启容器。
- 触发 AI。

验证：

- 给出 404 链接样本。

- 给出修复建议：改 link builder、加 fallback、改展示文案或加双链接。
