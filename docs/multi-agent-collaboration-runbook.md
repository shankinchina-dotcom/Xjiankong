# 多 Agent 协作操作手册

> 适用场景：长期项目、生产部署、跨 Agent 协作、弱模型执行、需要多轮验证和回滚的任务。
>
> 核心目标：把复杂任务拆成「计划 → 闸门 → 执行 → 回报 → 审计 → 下一闸门」，避免状态漂移、越权执行和无关改动混入。

## 一、基本原则

所有长期项目按以下模式执行：

```text
先写计划，再分闸门；
一次只放行一个闸门；
执行员只做被放行的动作；
审计员根据结果决定下一步；
生产、账号、密钥、部署永远单独确认。
```

禁止：

- Agent 自己连续执行多个阶段。
- 只口头说「继续」但没有明确范围。
- 没有计划文件就直接改生产。
- 执行 Agent 边做边改方案。
- 多个 Agent 同时改同一批文件或同一生产环境。

## 二、角色分工

### 1. 老板

职责：

- 决定目标。
- 批准方案。
- 放行高风险闸门。
- 判断最终效果是否接受。

以下动作必须单独确认：

- 上传生产文件。
- 修改 `.env`。
- 重启容器。
- 触发付费 AI。
- 修改 Cloudflare。
- 数据库迁移。
- 删除、回滚、覆盖文件。

### 2. 主控 Agent

职责：

- 看全局。
- 维护项目文档和计划。
- 拆任务、设闸门。
- 审查执行 Agent 的回报。
- 判断下一步能不能继续。
- 提供给执行 Agent 的精确指令。

主控 Agent 原则：

- 不直接相信执行报告，要核对逻辑。
- 不让执行 Agent 自己扩大范围。
- 不把「验证通过」写成「生产完成」。
- 发现计划不稳，先改计划，不继续执行。

### 3. 执行 Agent

职责：

- 按计划执行单一闸门。
- 回报事实，不做超范围判断。
- 遇到未知差异立即停止。
- 不擅自改计划、改生产、重启、触发 AI。

执行 Agent 必须输出：

```text
1. 执行了什么
2. 没执行什么
3. 结果是什么
4. 发现了什么差异
5. 是否触碰风险
6. 建议是否进入下一闸门
```

## 三、必须开计划模式的场景

以下场景必须开计划模式，并强制列 todo list：

- 跨多个文件的开发。
- 部署、生产同步、NAS、Docker、Cloudflare。
- 需要多个 Agent 配合。
- 需要连续执行超过 2 个步骤。
- 模型较弱，例如 DeepSeek、GLM、普通 Claude 子 Agent。
- 涉及状态追踪、回滚、验证。
- 任何「先做 A，再做 B，再验证 C」的任务。

计划模式最少要包含：

```text
1. 当前状态
2. 目标
3. 不做什么
4. 文件/系统范围
5. 分步骤 todo
6. 每步验证方式
7. 风险和回滚
8. 停止条件
9. 下一闸门
```

## 四、推荐工作流

### 阶段 1：主控写计划

主控 Agent 先写计划，不执行。

计划格式：

```text
# 任务计划

## 当前状态
- ...

## 目标
- ...

## 不在范围
- ...

## 硬约束
- ...

## 闸门

### 闸门 1：只读核验
允许：
- ...

禁止：
- ...

验证：
- ...

停止条件：
- ...

### 闸门 2：备份与上传
...

### 闸门 3：重启与免费验证
...

### 闸门 4：付费验证
...

### 闸门 5：公网验收
...
```

### 阶段 2：老板确认计划

老板只确认单一闸门：

```text
确认计划，可以进入闸门 1。
```

不要说「继续全部」，避免 Agent 理解成全自动执行。

### 阶段 3：执行 Agent 执行单一闸门

给执行 Agent 的标准指令：

```text
你只执行闸门 1，不执行后续步骤。

允许：
1. ...
2. ...

禁止：
1. ...
2. ...

完成后必须回报：
1. 执行命令摘要
2. 结果
3. 差异
4. 风险
5. 是否建议进入下一闸门

执行完停止，等待确认。
```

### 阶段 4：主控审计

执行 Agent 回报后，主控 Agent 做三件事：

```text
1. 判断报告是否完整。
2. 判断是否有未知差异或风险。
3. 给老板下一步建议。
```

输出格式：

```text
结论：可以 / 不可以进入下一闸门。

理由：
- ...

下一步建议：
- ...

给执行 Agent 的指令：
```text
...
```
```

### 阶段 5：老板再次确认

每个闸门都重复：

```text
老板确认 → 执行 Agent 执行 → 回报 → 主控审计 → 老板确认下一步
```

## 五、多 Agent 加入规则

如果以后加入更多 Agent，不要让他们平行乱跑。要按职能分开：

| 角色 | 适合任务 | 禁止 |
|---|---|---|
| 主控 Agent | 架构、计划、审计、最终判断 | 直接批量改生产 |
| 执行 Agent | 单一闸门执行 | 自行扩大范围 |
| UI Agent | 页面视觉、交互、截图反馈 | 改后端、部署 |
| 文档 Agent | 整理文档、会议记录、交接文档 | 判断生产放行 |
| 测试 Agent | 写测试、跑验证、整理失败原因 | 修代码后直接上线 |
| 运维 Agent | NAS、Docker、日志、备份 | 跳过确认重启/部署 |

多 Agent 协作必须有一个主控，不允许多个 Agent 同时当主控。

## 六、标准闸门模板

### 1. 只读核验闸门

```text
本轮只读，不修改任何文件或服务。

允许：
1. 查看状态
2. 查看日志
3. diff
4. inspect
5. checksum

禁止：
1. 上传
2. 覆盖
3. 删除
4. 重启
5. 触发付费调用
6. 修改 .env
7. 修改 Cloudflare

输出：
1. 当前状态
2. 差异列表
3. 白名单差异
4. 黑名单差异
5. 未知差异
6. 下一步建议
```

### 2. 上传闸门

```text
本轮只允许上传指定文件。

执行前：
1. 创建 BACKUP_DIR
2. 备份目标文件
3. 输出备份路径

允许：
1. 上传指定文件
2. 设置权限
3. diff / checksum 验证

禁止：
1. 重启
2. 触发任务
3. 改 .env
4. 改未列出的文件

输出：
1. 备份路径
2. 上传文件清单
3. 权限
4. diff / checksum 结果
5. 未执行事项
```

### 3. 重启闸门

```text
本轮只允许重启指定服务。

执行前：
1. 确认备份已存在
2. 确认配置已验证
3. 确认只重启指定服务

允许：
1. docker compose up -d <service>
2. docker ps
3. docker logs --tail

禁止：
1. 重启其他服务
2. prune / clean
3. 修改配置
4. 触发付费 AI

输出：
1. 容器状态
2. 使用镜像
3. 最近日志
4. 错误摘要
```

### 4. 付费验证闸门

```text
本轮会触发一次付费 AI 调用，执行前必须再次确认。

允许：
1. 手动触发一次任务
2. 查看日志
3. 查看输出报告
4. 估算费用

禁止：
1. 连续重试
2. 修改 prompt
3. 修改模型
4. 自动定时多次执行

输出：
1. 模型
2. 是否成功
3. 报告路径
4. 费用估算
5. 关键验收项
6. 失败原因
```

## 七、执行 Agent 必须使用 todo list

给弱模型的强制提示词：

```text
进入计划模式。

你必须先列 todo list，再执行。
每完成一项，标记状态。
一次只执行当前被批准的闸门。
遇到计划外差异，停止，不要自行修复。
不要跳到下一闸门。
不要为了完成任务猜测缺失信息。
```

todo 示例：

```text
- [ ] 读取项目规则
- [ ] 确认当前闸门范围
- [ ] 执行只读检查
- [ ] 汇总差异
- [ ] 判断是否命中停止条件
- [ ] 输出回报
- [ ] 停止等待确认
```

## 八、主控审计清单

主控 Agent 审计执行报告时检查：

```text
1. 有没有执行超范围动作？
2. 有没有遗漏禁止项说明？
3. 有没有混淆「本地验证」和「生产验证」？
4. 有没有把临时方案当正式方案？
5. 有没有隐藏未知差异？
6. 有没有触碰 .env、密钥、Cloudflare、数据库？
7. 有没有回滚路径？
8. 下一步是否真的依赖当前结果？
```

只要有一项不清楚，就不放行下一闸门。

## 九、推荐固定口径

### 1. 老板给主控 Agent

```text
先审这个执行报告，不要执行任何生产操作。
判断是否能进入下一闸门，并给我一段可以发给执行 Agent 的指令。
```

### 2. 主控给执行 Agent

```text
你只执行下面这个闸门，不执行后续步骤。
执行完成后停止，等待老板确认。
```

### 3. 执行 Agent 回报格式

```text
# 闸门执行报告

## 1. 执行范围
- ...

## 2. 实际执行
- ...

## 3. 结果
- ...

## 4. 差异
- 白名单：
- 黑名单：
- 未知：

## 5. 未执行事项
- ...

## 6. 风险
- ...

## 7. 下一步建议
- ...
```

## 十一、判断标准

DeepSeek、GLM 等弱模型必须强制计划模式和 todo list。

更强模型也建议强制，因为计划模式的价值不是让模型更聪明，而是把隐含状态外部化，方便老板、主控 Agent、执行 Agent 三方审计。

最终标准：

```text
没有计划，不执行；
没有闸门，不生产；
没有验证，不宣布完成；
没有回滚，不上线。
```
